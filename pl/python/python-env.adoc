

== 📦 工具详解与对比

[cols="1,2,3,3,3", options="header"]
|===
| 工具 | 核心定位 | 优点 | 缺点 | 适用场景

| pyenv
| ✅ 解释器版本管理（事实标准）
| • 轻量精准管理多 Python 版本+• .python-version 项目级锁定+• 插件生态丰富（pyenv-virtualenv 联动虚拟环境）
| • 本身不创建虚拟环境（需插件）+• 编译安装依赖系统库（可能失败）+• Windows 需 pyenv-win（体验稍弱）
| 需频繁切换 CPython 主版本的开发者

| venv
| ✅ 标准库虚拟环境
| • 无需安装（Python 3.3+ 内置）+• 简单安全，与语言版本绑定+• 零外部依赖
| • 功能基础（无环境列表/快捷切换）+• 无法管理解释器版本+• 旧版 Python（<3.3）不支持
| 简单脚本/教学场景；追求最小依赖；CI 环境

| virtualenv
| ✅ 增强虚拟环境
| • 支持旧版 Python（2.7+）+• 创建速度更快（缓存机制）+• 配合 virtualenvwrapper（workon 快速切换）
| • Python 3.3+ 场景下优势减弱+• 需额外安装（pip install virtualenv）
| 需支持旧 Python 版本；依赖 virtualenvwrapper 工作流

| conda
| ✅ 跨语言环境管理
| • 二进制包预编译（解决 numpy/tf/cuda 安装难题）+• 环境彻底隔离（含非 Python 依赖）+• 跨平台一致体验
| • 体积庞大（Miniconda 50MB+）+• 包版本可能滞后 PyPI+• 与 pip 混用易冲突（需严格流程）
| 数据科学/ML；需管理 R/CUDA 等非 Python 依赖；跨平台团队

| poetry
| ✅ 项目+依赖+环境一体化
| • pyproject.toml 统一配置（PEP 621）+• 依赖解析精准（避免冲突）+• 内置环境管理（自动创建/激活）+• 发布流程简化
| • 学习曲线较陡+• 虚拟环境默认存全局（需 poetry config virtualenvs.in-project true 放项目内）+• 与传统 requirements.txt 工作流不兼容
| 新项目开发；重视依赖可重现性与发布流程；开源项目

| uv 🚀
| ✅ 下一代全能工具（Rust 编写）
| • 极速：安装包比 pip 快 10-100 倍+• 命令统一：uv venv/uv pip/uv project+• 兼容现有生态（requirements.txt, pyproject.toml）+• 内存占用低+• CI/CD 友好
| • 较新（2024.03 发布），企业级验证中+• uv python install 标注 experimental+• 边缘场景兼容性待社区验证
| 追求速度的新项目；CI/CD 优化；愿意尝鲜；现代工作流倡导者

| pipenv
| ⚠️ 历史工具
| • 曾获 PyPA 推荐（2018）+• Pipfile/Pipfile.lock 管理依赖
| • 性能慢（依赖解析耗时）+• 维护活跃度低（2023 后更新稀疏）+• 与 poetry 功能重叠但体验较差
| 仅限维护遗留项目；新项目强烈不推荐
|===
