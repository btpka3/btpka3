== Playwright MCP 工作原理

[source,plain]
----
  Claude Code  ──(stdio/SSE)──▶  Playwright MCP Server  ──(CDP)──▶  Chromium 浏览器
     (客户端)                      (Node.js 进程)                    (无头/有头)
----

1. Claude Code 是 MCP 客户端，通过 stdio 或 SSE 协议与 MCP Server 通信
2. Playwright MCP Server 是一个 Node.js 进程，暴露一组工具（navigate、screenshot、click、evaluate 等）
3. 我调用这些工具时，MCP Server 通过 Chrome DevTools Protocol (CDP) 操控浏览器执行操作
4. 截图会以 base64 图片返回给我，我能直接"看到"页面渲染结果

本质上就是给我装了一双能看浏览器的"眼睛"和能操作浏览器的"手"。

== browser/浏览器

如果安装了  Playwright MCP  或者    Puppeteer MCP  ， 能复用我个人在用的 chrome 浏览器 的窗口吗，这样的话就能复用一些网站的登录态，保证cookie 等有效可用。
[source,plain]
----
⏺ 可以，有两种方式：

  方式一：连接已有 Chrome 实例（推荐）

  先用远程调试端口启动你日常的 Chrome：

  # 关闭所有 Chrome，然后用这个命令启动
  open -na "Google Chrome" --args --remote-debugging-port=9222

  然后配置 MCP 连接到它：

  Playwright MCP：
  {
    "mcpServers": {
      "playwright": {
        "command": "npx",
        "args": ["@playwright/mcp", "--cdp-endpoint", "http://127.0.0.1:9222"]
      }
    }
  }

  Puppeteer MCP：
  {
    "mcpServers": {
      "puppeteer": {
        "command": "npx",
        "args": ["-y", "@modelcontextprotocol/server-puppeteer"],
        "env": {
          "PUPPETEER_BROWSER_URL": "http://127.0.0.1:9222"
        }
      }
    }
  }

  这种方式直接复用你正在用的 Chrome 窗口，所有 cookie、登录态完全一致。

  方式二：复用 Chrome 用户数据目录

  让 MCP 启动新 Chrome 进程，但指向你已有的 Profile 数据：

  # macOS 上你的 Chrome 数据目录通常在：
  ~/Library/Application Support/Google/Chrome

  但有一个硬限制： 同一个 user-data-dir 不能同时被两个 Chrome 进程使用。所以要么关掉你的 Chrome，要么复制一份 Profile（cookie 是快照，不会实时同步）。

  对比

  ┌───────────────────┬────────────────────────────────┬──────────────────────┐
  │                   │      方式一：连接已有实例      │ 方式二：复用数据目录 │
  ├───────────────────┼────────────────────────────────┼──────────────────────┤
  │ Cookie/登录态     │ ✅ 实时共享                    │ ⚠️  启动时的快照     │
  ├───────────────────┼────────────────────────────────┼──────────────────────┤
  │ 能否同时用 Chrome │ ✅ 共用同一进程                │ ❌ 需关闭你的 Chrome │
  ├───────────────────┼────────────────────────────────┼──────────────────────┤
  │ 配置复杂度        │ 低                             │ 中                   │
  ├───────────────────┼────────────────────────────────┼──────────────────────┤
  │ 风险              │ MCP 操作会影响你正在用的标签页 │ 无互相干扰           │
  └───────────────────┴────────────────────────────────┴──────────────────────┘

  ---
  建议方式一。唯一需要改变的习惯是每次启动 Chrome 时带上 --remote-debugging-port=9222。可以做成一个 alias：

  # 加到 ~/.zshrc
  alias chrome='open -na "Google Chrome" --args --remote-debugging-port=9222'

  这样以后输入 chrome 就自动带调试端口启动，Claude 就能直接连上去操作你的浏览器、复用你的登录态。
----

== 扩展

1. Hooks（生命周期钩子）
在 settings.json 中配置，可在特定事件触发时执行 shell 命令：

** SessionStart / SessionEnd — 会话开始/结束
** PreToolUse / PostToolUse / PostToolUseFailure — 工具调用前/后/失败
** UserPromptSubmit — 用户提交消息时
** PermissionRequest — 权限请求时
** SubagentStart / SubagentStop — 子 agent 启停
** Stop — 停止时

2. MCP Servers（模型上下文协议）
  通过 claude mcp add 或 ~/.claude.json 配置，支持 stdio 和 http 两种传输方式。
  MCP 是最强大的扩展点，可以给 Claude 增加任意工具能力。

3. CLAUDE.md（指令文件）

  多级指令体系，自动加载：
** ~/.claude/CLAUDE.md — 全局个人指令
** 项目根目录 CLAUDE.md — 项目级指令
** .claude/CLAUDE.md — 项目私有指令

4. Plugins（插件系统）
通过 claude plugin install/enable/disable 管理：

** 支持 marketplace 安装
** 支持 --plugin-dir 加载本地插件

5. Custom Agents（自定义 Agent）

** --agents 参数传入 JSON 定义自定义 agent
** --agent 指定当前会话使用的 agent
** claude agents 管理后台 agent

6. Skills（技能/斜杠命令）
内置的 /init、/review、/simplify 等，也支持通过插件注册自定义 skill。

7. Settings（配置体系）

三层配置：
** ~/.claude/settings.json — 用户级
** .claude/settings.json — 项目级
** .claude/settings.local.json — 项目本地（不提交）

  可配置：权限白名单、环境变量、hooks、MCP servers 等。

8. Memory（自动记忆）

~/.claude/projects/<path>/memory/ 下的文件系统，跨会话持久化记忆。

9. CLI 管道模式

--print + --output-format stream-json + --input-format stream-json 支持程序化调用，可嵌入 CI/CD 或其他工具链。

== mcp/tool 太多

[source, plain]
----
1. Skills 机制(这是唯一真正的 lazy load)
- skill 元数据在系统提示里只占一行(描述),内容只在 Skill 工具调用时才加载
- 一个 skill 内部可以包含很多 sub-tool 调用逻辑、文档、prompt 模板
- 如果你想做"按需 MCP",最佳实践是把一组 MCP 包装成一个 skill:skill 描述里说"调用此 skill 后会暴露 xx 工具",触发后再展开

2. --mcp-config 切换 profile(启动时,非运行时)

# 翻译类轻量任务
claude --mcp-config ~/.claude/profiles/minimal.json --strict-mcp-config

# 业务开发任务
claude --mcp-config ~/.claude/profiles/aone-dev.json

--strict-mcp-config 会忽略全局/项目级 mcpServers,只用指定 config。

3. settings 层级白名单

~/.claude/settings.json:
{
  "enableAllProjectMcpServers": false,
  "enabledMcpjsonServers": ["code", "aone-mix"],
  "disabledMcpjsonServers": ["aliDingDoc", "sunfire-mcp", "arthas"]
}

依然是会话启动时生效,不是运行时。

4. Subagent 隔离工具上下文

- 主 agent 配置最小工具集
- 通过 Agent 工具派发到 subagent
- subagent 在自己的 agent definition (.claude/agents/*.md) 里声明 tools: 字段限定可用工具
- 效果:重型工具只活在 subagent 自己的上下文里,主 agent 的 200K 不被污染

5. 运行时关闭(伪)

- /mcp 斜杠命令可以查看 MCP server 状态、断连个别 server
- 但断连不会从已建立的会话里清除已注入的 tool schema(因为后续请求 tools 数组的构建依赖启动时快照)
- 真要释放,只能重启 CLI

6. MCP Gateway 模式(社区/自建)

自己写一个"门面" MCP server:
- 只对外暴露 2 个 meta tool:list_tools(category) 和 invoke_tool(name, args)
- 实际几百个底层工具的 schema 不进上下文,需要时由 meta tool 返回
- 代价:模型失去 schema 的强类型约束,容易出错;复杂参数需要 meta tool 自己做校验
- 这是目前业界(包括 Cursor/Cline 圈)讨论较多的折中方案
----

== skills

link:https://code.claude.com/docs/zh-CN/skills[使用 skills 扩展 Claude]


自定义命令已合并到 skills 中。 .claude/commands/deploy.md 中的文件和 .claude/skills/deploy/SKILL.md 中的 skill 都会创建 /deploy 并以相同的方式工作。你现有的 .claude/commands/ 文件继续工作。

故自定义 command 现在就是 自定义 skill

将 SKILL.md 保持在 500 行以下.

SKILL.md 中的 关键配置选项（YAML frontmatter）

[source, yaml]
----
name: ""                            # skill/command 的名称,最多 64 个字符
description :  ""                   # 描述命令用途，Claude 据此判断是否自动触发 , 和 when_to_use 一起最多 1536 个字符
when_to_use : ""                    # 何时应该调用该 skill 的额外上下文，例如触发短语或示例请求。, 和 description 一起最多 1536 个字符
argument-hint : "[filename] [format]" # 自动完成期间显示的提示，指示预期的参数
disable-model-invocation: true      # 禁止 Claude 自动调用，只允许你手动 /command 触发
user-invocable: false               # 在 / 菜单中隐藏，仅 Claude 可自动调用
allowed-tools : []                  # 预授权工具，如 Read Grep Bash(git *)
context: fork                       # 在隔离子 Agent 中运行
arguments: []                       # 声明参数，如 arguments: [issue, branch]，内容中用 $issue $branch 引用
model: ""
effort: high                        # 当此 skill 处于活动状态时的工作量级别任务难度，low, medium, high
agent: ""                           # 当设置 context: fork 时要使用的 subagent 类型。
hooks: {} #限定于此 skill 生命周期的 hooks
paths: [""] # Glob 模式，限制何时激活此 skill。接受逗号分隔的字符串或 YAML 列表。
shell: # 用于此 skill 中 !`command` 和 ```! 块的 shell
----
