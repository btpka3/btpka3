* Puppeteer MCP

Playwright 由 Microsoft 开发，支持 Chromium、Firefox、WebKit 三大引擎，原生支持 headless 模式。API 设计对 AI agent 非常友好，提供了自动等待、元素定位、网络拦截等丰富能力。


qoder 提供了浏览器的插件，其基于 Playwright 框架实现 相关浏览器自动化测试能力。

CDP : Chrome Dev Tools Protocol


== 竞品

* Puppeteer： 是 Google 官方维护的 Chrome DevTools Protocol 封装，专注于 Chromium 系列浏览器。API 简洁，headless 模式是默认行为。适合只需要 Chrome/Edge 的场景。

* Selenium WebDriver: 是最老牌的框架，通过 W3C WebDriver 协议控制浏览器。支持浏览器最广（IE、Safari、Firefox、Chrome 等），但 API 偏底层，自动等待能力弱，对 AI agent 不够友好。

* Cypress: 直接在浏览器内运行测试，执行速度快，调试体验好。但它不是真正的 headless 浏览器（是在浏览器内注入测试代码），且只支持 Chromium 和 Firefox，跨域/多标签页场景支持有限。

== 遇到需要用户确认或介入的场景

这是 AI agent 自动化中最常见的挑战，有几种处理方式：

* 第一种：预判并自动化所有分支。 在测试脚本中提前处理可能的弹窗、确认对话框。比如 Playwright 中可以监听 dialog 事件并自动处理：

[source,js]
----
page.on('dialog', dialog => dialog.accept());
----

* 第二种：设置断点/暂停机制。 Playwright 和 Puppeteer 都支持 page.pause()，测试执行到该点时暂停，打开浏览器让用户手动操作，完成后继续。适合调试和半自动化场景。

* 第三种：人机协作模式。 AI agent 执行到需要确认的步骤时，截图发送给人类，人类确认后返回指令，agent 继续执行。这在复杂审批流、验证码、多因素认证等场景中很常见。

* 第四种：Mock/Stub 绕过。 对于验证码、短信验证等不适合自动化的环节，在测试环境中 Mock 掉这些功能。这是 CI/CD 中最推荐的做法。

* 第五种：环境变量控制确认行为。 通过环境变量或配置文件控制测试在 headless 模式下自动确认，在有头模式下暂停等待人工确认：

[source,js]
----
const shouldPause = process.env.INTERACTIVE === 'true';
if (shouldPause) await page.pause();
----

        "command": "npx",
        "args": ["@playwright/mcp"]
      }
    }
  }

  Puppeteer MCP：
  {
    "mcpServers": {
      "puppeteer": {
        "command": "npx",
        "args": ["-y", "@modelcontextprotocol/server-puppeteer"]
      }
    }
  }
