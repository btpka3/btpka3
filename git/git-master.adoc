Spring Framework 等大型 Java 开源项目在维护多版本（如 2.x、3.x）时，通常采用一套成熟的分支管理策略和工程化流程来平衡功能迭代、漏洞修复与测试验证。以下是主流实践的详细拆解：

== 一、Git 分支管理策略：多主线并行 + 定向合并
大型项目通常不依赖单一主干分支，而是为每个活跃大版本维护独立的主线分支，例如：

* main / master → 对应最新稳定版（如 6.x）
* 5.3.x → Spring Framework 5.3 系列维护分支
* 4.3.x → 历史长期支持分支

核心规则：

* 漏洞/安全修复：优先在最老仍受支持的版本分支上修复，再通过 git cherry-pick 或手动合并到新版本分支。这确保所有受影响版本都能及时获得补丁。
* 新功能开发：仅在最新主线分支进行，不会向后移植到旧版本，以避免破坏 API 兼容性或引入不必要的复杂度。
* 分支生命周期：每个大版本有明确的 EOL（End of Life）时间表。一旦停止维护，该分支将归档，不再接受任何提交。


== 二、变更应用策略：如何避免“重复开发两遍”？
当主要 API 已发生显著变化时，项目通常通过以下机制降低重复工作量：

✅ 1. 共享核心模块（Shared Core）
将跨版本通用的底层逻辑抽离为独立模块，例如：

spring-core、spring-beans 中的基础工具类、类型转换、资源加载等
这些模块在不同版本间保持 API 兼容，只需维护一份代码，通过条件编译或抽象层适配差异
✅ 2. 向后兼容层（Compatibility Layer）
在新版本中提供旧 API 的兼容实现，例如：

使用 @Deprecated 标注旧方法，内部委托给新实现
通过适配器模式（Adapter Pattern）桥接新旧接口，减少业务层改动
✅ 3. 自动化代码迁移工具
Spring 官方提供 spring-boot-maven-plugin 等工具，可自动扫描并提示 API 变更点。对于大规模重构，团队会编写脚本批量替换过时调用。

注意：
安全漏洞修复通常涉及底层组件（如 XML 解析、反序列化、权限校验），这些逻辑在多数版本中结构相似，因此 cherry-pick 成功率较高。
功能性新特性（如响应式编程、虚拟线程支持）因架构差异大，一般不会跨大版本移植，而是作为新版本的独占能力。


== 🔹三、自动化测试与验证体系
为确保多版本变更的正确性，Spring 等项目构建了分层测试 pipeline：

=== 1. 单元测试覆盖核心契约

每个模块配备高覆盖率单元测试，重点验证公共 API 的行为一致性
使用 JUnit 5 + Mockito，结合参数化测试覆盖边界条件
=== 🔗2. 集成测试验证组件协同

启动嵌入式容器（如 Tomcat、Netty）模拟真实运行环境
验证 Bean 生命周期、事务管理、AOP 代理等关键机制是否正常
=== 3. 兼容性测试矩阵

针对不同 JDK 版本（8、11、17、21）、构建工具（Maven/Gradle）、操作系统进行交叉测试
使用 GitHub Actions 或 Jenkins 定义多维测试矩阵，自动触发各版本分支的 CI 流程
=== 4. 安全专项测试
集成 SAST（静态应用安全测试）工具扫描已知漏洞模式
对 CVE 相关修复编写回归测试用例，防止问题复发
使用 OWASP Dependency-Check 检查第三方依赖风险


=== 🔄5. 发布前验证

生成 SNAPSHOT 版本供社区早期试用
执行端到端场景测试（如完整 Web 应用部署、微服务通信）
最终通过 maven-release-plugin 或 Gradle 发布插件完成签名、上传至 Maven Central

== 🔹四、典型工作流示例（以修复 CVE 为例）

* 安全团队报告某版本存在反序列化漏洞
* 维护者在对应版本分支（如 5.3.x）创建 hotfix 分支
* 修复代码后，运行该分支全部测试套件
* 使用 git cherry-pick 将 commit 应用到 6.0.x 及 main
* 若 API 差异导致冲突，则手动调整实现，但保留相同的安全逻辑
* 所有目标分支 CI 通过后，同步发布补丁版本（如 5.3.31、6.0.12）
* 更新安全公告，通知用户升级
