
== JDK
=== JDK 8
* Lambda 表达式与 Stream API
** 并行流：利用 parallelStream() 轻松实现并行处理，充分利用多核 CPU
** 惰性求值：Stream 操作链按需执行，避免不必要的中间计算
** 短路操作：findFirst()、anyMatch() 等可在满足条件时提前终止

* 永久代

** 元空间：将类元数据从堆内存移至本地内存，减少 Full GC 频率
** 可调整大小：动态调整元空间大小，避免 OutOfMemoryError: PermGen

* 改进的 HashMap
** 红黑树优化：冲突超过阈值时使用红黑树替代链表，查询复杂度从 O(n) 降至 O(log n)
** 性能提升：在大量哈希冲突场景下显著提高性能


=== JDK 9-11
* G1 GC 改进
** JEP 248：G1 成为默认垃圾收集器
** JEP 304：可切换的垃圾收集器：运行时动态切换 GC 算法

* 字符串压缩
** Compact Strings：使用 byte[] 替代 char[] 存储纯 ASCII 字符串，节省内存

* 应用类数据共享
** AppCDS：跨进程共享类元数据，减少启动时间和内存占用


* GraalVM JIT 编译器, JDK 9+ 可选，提供激进优化技术
* VarHandle：JDK 9，提供比反射更高效的字段访问

=== JDK 12-17
* ZGC (JDK 11 引入，12+ 持续优化)
**  低延迟 GC：停顿时间不超过 10ms
** 支持大堆内存：TB 级堆内存管理
** JDK 15 生产就绪


* Shenandoah GC (JDK 12)
** 并发整理：GC 与应用线程并发执行
** 超低停顿：停顿时间与堆大小无关

* 记录类型
** JDK 14/15/16：轻量级数据载体，自动生成构造器、equals、hashCode 等
** 性能优势：减少样板代码，提高运行效率

* 模式匹配
** JDK 16/17：简化 instanceof 检查和类型转换
** 性能提升：编译器优化，减少类型检查开销

=== JDK 18-21

* 虚拟线程 (JDK 21)
**  轻量级线程：百万级虚拟线程，大幅提升并发吞吐量
** 阻塞成本极低：传统阻塞操作不再成为性能瓶颈

* 向量 API (JDK 16-21 持续演进)
** SIMD 指令支持：利用 CPU 向量指令加速计算密集型任务
** 性能倍增：矩阵运算、图像处理等场景性能显著提升

* 结构化并发
** 并发任务管理：简化并发编程，减少上下文切换开销

* 分代 ZGC (JDK 21)
** 性能提升：针对年轻代和年老代分别优化，降低 CPU 开销
** 内存效率：减少内存占用，提高吞吐量

== Leyden

JEP 483: Ahead-of-Time Class Loading and Linking


* 不是完全编译成 native code：与 GraalVM Native Image 不同，它不是将整个应用编译成原生可执行文件，而是将类以已加载和已链接的状态缓存在 JVM 中

* 使用 AOT Cache：
** 第一次运行时（training run）记录类的加载信息
** 创建 AOT cache 文件（.aot）
** 后续运行时直接从 cache 中读取已加载和链接的类

* 基于 CDS（Class Data Sharing）：这是 CDS 技术的演进，不仅仅是解析类文件，还包括了加载和链接

[source,shell]
----
# 步骤 1: 训练运行
java -XX:AOTMode=record -XX:AOTConfiguration=app.aotconf -jar my-app.jar

# 步骤 2: 创建缓存
java -XX:AOTMode=create -XX:AOTConfiguration=app.aotconf -XX:AOTCache=app.aot -jar my-app.jar

# 步骤 3: 使用缓存运行
java -XX:AOTCache=app.aot -jar my-app.jar
----

限制
目前只支持 JDK 内置的 classloader 加载的类（从 classpath、module path 加载的类），不支持用户自定义 classloader 加载的类。

这个特性是 Project Leyden 的第一个交付成果，后续还会有更多优化（如 AOT Code Compilation 等）。
