maven extension vs. plugin

Extension 是给 Maven "装新的器官"，Plugin 是让 Maven "做新的动作"。

Extension（扩展） 的核心作用是改变 Maven 自身的构建行为和能力，它在构建生命周期的极早期就被加载，主要用途包括：

  • 引入新的打包类型 — 例如支持 .war、.ear 或其他自定义格式

  • 提供自定义的 artifact handler 或 lifecycle mapping — 让 Maven 识别新的项目类型

  • 扩展 Repository 传输协议 — 例如支持 s3://、gcs:// 等非标准协议

  • 修改类加载器行为 — 在构建开始前注入自定义的类或资源

维度 Extension Plugin

加载时机

构建极早期，项目模型解析之前

生命周期阶段到达时

作用

扩展 Maven 核心能力（新协议、新打包类型等）

执行具体构建任务（编译、测试、打包等）

声明位置

<build><extensions>

<build><plugins>

是否可绑定 goal

否，没有 goal 概念

是，通过 goal 绑定到 phase

典型场景

wagon 传输扩展、自定义 lifecycle mapping

编译、测试、代码生成、部署

注意事项

mvn 命令直接执行的 goal 需要绑定 maven 生命周期，否则缓存失效。

示例命令: mvn clean package docker:build

可以参考下面的示例，将其绑定 maven 生命周期

<plugin>
    <groupId>com.spotify</groupId>
    <artifactId>docker-maven-plugin</artifactId>
    <version>${docker.plugin.version}</version>
    <executions>
        <execution>
            <id>my-build</id>
            <phase>package</phase>
            <goals>
                <goal>build</goal>
            </goals>
        </execution>
    </executions>
</plugin>

解析

<build><artifact>
<build><attachedArtifacts>                          # 记录了配置的 附属产物
<build><executions><execution>                      # 记录各个maven 插件的相关信息，和属性值，
                                                    # tracked=true 表示在 build-cache-report.xml 的 reconciles/reconcile 里有声明，值有变化就缓存失效。
                                                    # tracked=false 则表示值的变动不会触发缓存失效
<build><executions><execution><properties><property> # 记录的是 mojo 字段的 toString() 的值，但如果字段类型是复杂POJO对象，且没有重写 toString(),
                                                     # 记录的值则会 是 `${类名}@${hashcode}` , 比如 spotless-maven-plugin 的 com.diffplug.spotless.maven.SpotlessCheckMojo 的 `java` 字段记录的值如下：
                                                     # <property name="java" tracked="false">com.diffplug.spotless.maven.java.Java@188ae8d2</property>
                                                     # 故这中字段则不能在 maven-build-cache-config.xml 设置到 reconciles/reconcile 里。
<build><projectsInputInfo><items><item type="pom">  # 记录了 effective pom ，故如果 maven profile 影响了依赖的jar包,plugin的版本，会触发这里的变动
<build><projectsInputInfo><items><item type="file"> # 通常记录了 src 目录下 相关源文件。

解析

<cache><configuration>
<cache><configuration><attachedOutputs>             # target/ 下哪些的目录（非标准产物）需要作为 添加额外需要缓存的内容, 比如缓存 classes 目录、jacoco 报告、生成的代码等，通过 目录路径 + glob 过滤， 然后将目录 zip 打包后作为附加产物缓存
<cache><configuration><attachedOutputs><dirNames>dirName>
相对于 target/ 目录的
配置                                    实际路径
<dirName>classes</dirName>              ${project.basedir}/target/classes
<dirName glob="jacoco.xml"></dirName>   ${project.basedir}/target/jacoco.xml（空值表示 target 目录本身）
<dirName>../src/main/javagen</dirName>  ${project.basedir}/src/main/javagen（跳出 target 目录）

<cache><input>
<cache><input><plugins><plugin>                     # 控制插件级别的输入计算规则配置
<cache><input><plugins><plugin><effectivePom><excludeProperties>
会影响 buildinfo.xml 中 <item type="pom"> 的内容（即影响  effective pom  的内容）, 被排除的 属性不会作为 effective pom 的内容，不影响缓存命中。

<cache><input><plugins><plugin><dirScan>
控制插件配置中引用的目录如何扫描。某些插件的配置属性指向一个目录，但只需要扫描其中的特定文件。
比如 protoc-maven-plugin 的 protoBaseDirectory 指向一个目录，但只需要扫描 .proto 文件：
<plugin artifactId="protoc-maven-plugin">
    <dirScan mode="auto">
        <tagScanConfigs>
            <tagScanConfig tagName="protoBaseDirectory" recursive="false" glob="{*.proto}"/>
        </tagScanConfigs>
    </dirScan>
</plugin>


<cache><input><plugins><plugin><executions><execution>
针对 maven 插件某个 execution 的更新粒度配置，配置基本上同上。



<cache><output>                                     # Maven 的附加 artifacts 。 正则表达式 pattern 匹配 artifacts 文件名，
<cache><output><includes><include>
<cache><output><excludes><exclude>

<cache><executionControl><runAlways>                # 始终执行执行的 maven 插件，execution，goal
<cache><executionControl><runAlways><plugins>
<cache><executionControl><runAlways><executions>
<cache><executionControl><runAlways><goalsLists>
<cache><executionControl><ignoreMissing>            # 哪些 maven 插件，execution，goal 可以被完全忽略（即不影响构建产物）
<cache><executionControl><ignoreMissing><plugins>
<cache><executionControl><ignoreMissing><executions>
<cache><executionControl><ignoreMissing><goalsLists>

<cache><executionControl><reconcile><plugins><plugin>
<cache><executionControl><reconcile><plugins><plugin><reconciles><reconcile>
设置 要追踪 maven 插件的 mojo 类的哪些 java 字段名的值是否变化。
未声明的 字段名，其值的变化不影响缓存是否失效。
对应于 buildinfo.xml  <build><executions><execution> 下 tracked=true 的元素。
● propertyName 是对应 goal 的 Mojo 类的字段名。
注意：不完全等同于 pom.xml 中该 maven 插件 <configuration> 下的 xml 元素名的值，但有一定的映射关系。

比如 mvn test命令对应的 maven-surefire-plugin 插件的 org.apache.maven.plugin.surefire.SurefireMojo 类， 上面示例就只得是跟踪 SurefireMojo#skip 字段（声明在父类中）的值。
但如果值是 "true" 时，则不会触发缓存失效。因为此次没执行 单元测试，不会影响构建产物的生成。
● skipValue ： Mojo 的字段的哪个值可以忽略，不会影响mvn 构建产物。
● defaultValue : 说明其默认值。
SurefireMojo 的 skip 字段有通过 @Parameter(property = "maven.test.skip", defaultValue = "false") protected boolean skip; 注解声明默认值，故这种可以不声明 defaultValue.
但如果字段类型对象型，没有通过 @Parameter注解设置默认值，且 该Mojo内部程序逻辑 是将 null 和 "xxx" 认为等同的 时，
如果仅仅声明 <reconcile propertyName="myProp" /> , 未说明其默认值。
实际首次未设置时， maven-build-cache-extension 则首次计算该字段值是 会认为是 null，
后面有更新成新值 "xxx",  maven-build-cache-extension 会认为其值变更成了 "xxx", 就会触发缓存失效。
此时就可以通过  声明 defaultValue 来避免了 ： <reconcile propertyName="myProp" defaultValue="xxx" />。
● 文本值 ： 根据源码解析，虽然允许 <reconcile propertyName="myProp">xxxx</reconcile>设置元素的文本块内容，但这部分程序里还没用到。

<cache><executionControl><reconcile><plugins><plugin><logs>、<nologs>
哪些 Mojo 插件的属性值是否要被记录到 buildinfo.xml 的 <build><executions><execution><properties> 中， 不参与 缓存命中判断，只控制"记不记"。
如果该属性已经用 <reconciles><reconcile> 配置了，则一定会记录到  buildinfo.xml 中。

reconciles/reconcile

Maven 构建缓存的核心逻辑是：如果项目输入（源码、依赖、插件配置等）没有变化，就复用之前的构建产物。但有些插件属性会受命令行参数影响，例如 -Dmaven.compiler.source=17。这类属性的值在不同构建之间可能不同，但缓存键（cache key）计算时可能并未完全捕捉到这些差异。

reconcile 机制就是用来解决这个问题的——它会在缓存恢复时，对比当前构建的插件属性值与缓存中记录的值，如果发现关键属性不一致，就判定缓存不可用，从而触发重新构建。

<reconcile logAllProperties="true">
  <plugins>
    <plugin groupId="org.apache.maven.plugins"
            artifactId="maven-compiler-plugin"
            goal="compile">
      <reconciles>
        <reconcile propertyName="source"
                   skipValue="${maven.compiler.source}"
                   defaultValue="1.8"/>
        <reconcile propertyName="target"
                   skipValue="${maven.compiler.target}"
                   defaultValue="1.8"/>
      </reconciles>
    </plugin>
  </plugins>
</reconcile>

每个 <reconcile>` 子元素有三个属性：

  • propertyName（必填）：要追踪的插件属性名称，对应插件 Mojo 中的参数名。

  • skipValue（可选）：当属性值等于 skipValue 时，跳过对该属性的比对。 这通常用于处理未解析的占位符——如果属性值仍然是 ${maven.compiler.source}` 这样的表达式（说明用户没有在命令行或 POM 中显式设置）， 就跳过比对，因为此时值没有实际意义。

  • defaultValue（可选）：当属性值为 null（即插件未显式配置该属性）时，使用的默认值参与比对。 这确保即使属性未在 POM 中声明，也能有一个基准值来判断缓存是否匹配。
