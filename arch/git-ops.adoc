== 什么是 GitOps？
GitOps 是一种以 Git 作为单一事实来源（Single Source of Truth）的现代应用交付和运维范式。它将基础设施和应用配置的整个生命周期管理都纳入版本控制系统，通过声明式配置和自动化实现持续交付。


核心原则



* 声明式配置:	     使用 YAML/JSON 等声明系统"应该是什么状态"，而非"如何达到该状态"
* Git 为唯一真相源:	所有基础设施和应用配置都存储在 Git 仓库中
* 自动化同步:	    系统自动将集群实际状态与 Git 中声明的期望状态对齐
* 变更审计可追溯:	所有变更通过 Pull/Merge Request 进行，保留完整历史记录

## vs. 运行时动态配置

Q: 运行态有大量可运行时调整的配置的场景，是不是不使用该场景？
A: 不是不使用，而是需要分层架构设计。问题提到场景 （Web 控制台配置 → 远程加载资源 → 动态开启功能）完全适合运行时动态配置，不应强行塞进 GitOps。

### 核心区分：什么该进 Git，什么不该进？

[cols="1,1,1",options="header"]
|===
| 维度
| GitOps 管理（静态/半静态）
| 运行时动态配置

| 变更频率
| 低频（天/周级）
| 高频（秒/分钟级）

| 影响范围
| 全局/结构性变更
| 局部/行为性调整

| 回滚方式
| `git revert`
| 控制台撤销/热更新

| 典型例子
| 镜像版本、副本数、路由规则
| 功能开关、限流阈值、UI 文案
|===


总结：何时用 GitOps，何时不用？
* 使用 GitOps：

** 基础设施即代码（IaC）
** 应用部署清单（Deployment/StatefulSet）
** 网络策略、RBAC、证书
** 需要严格审计和合规的配置

* 不使用 GitOps（改用动态配置）：

** 功能开关（Feature Flags）
** 业务规则/阈值调整
** A/B 测试、灰度发布策略
** 需即时生效且频繁变更的参数
