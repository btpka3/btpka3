= Go / Java / Rust 原生 CLI 开发对比

[cols="2,3,3,3", options="header"]
|===
| 对比维度 | Go | Java (GraalVM Native Image) | Rust

| 开发语言
| Go
| Java
| Rust

| 编译模型
| 静态编译，内置 Runtime
| AOT 提前编译 (Native Image)
| 静态编译，零 Runtime

| 产物形态
| 单文件原生可执行文件
| 单文件原生可执行文件
| 单文件原生可执行文件

| 典型产物体积 (小型 CLI)
| 2–5 MB
| 10–20 MB
| 0.5–2 MB

| 典型产物体积 (中型 CLI)
| 8–25 MB
| 15–50 MB
| 1–5 MB

| 运行时依赖
| 零依赖 (除系统 libc)
| 零依赖 (已静态链接 Substrate VM)
| 零依赖 (完全静态链接)

| 交叉编译难度
| 极低 (GOOS/GOARCH 环境变量即可)
| 中等 (需对应平台 GraalVM 或容器化构建)
| 低 (cargo-cross / zigbuild)

| 启动速度
| 快 (毫秒级)
| 极快 (毫秒级，无 JVM 预热)
| 极快 (微秒到毫秒级)

| 内存占用
| 较低 (Runtime 有 GC 开销)
| 低 (无 JVM 堆，但有 Substrate VM 开销)
| 极低 (无 GC，手动/RAII 内存管理)

| 性能表现
| 良好 (适合 IO 密集型)
| 良好 (接近原生，但受 AOT 限制)
| 极佳 (接近 C/C++，适合计算密集型)

| CLI 框架生态
| 丰富 (cobra, urfave/cli, flag)
| 较少 (picocli 配合 Native Image 较好)
| 丰富 (clap, argh, bpaf)

| 标准库完备度
| 高 (net/http, encoding/json, os, sync 等)
| 高 (JDK 生态庞大)
| 高 (std 覆盖文件、网络、线程、集合等)

| 异步/并发模型
| Goroutine + Channel (CSP 模型)
| CompletableFuture / Virtual Threads (JDK 21+)
| async/await + Tokio (基于 Futures)

| 内存安全
| GC 自动回收 (有内存泄漏可能)
| GC 自动回收 (Native Image 也有 GC)
| 编译期借用检查 (无 GC，零成本安全)

| 错误处理风格
| 显式 error 返回值，简单直接
| 异常 (Exception) + Optional/Result
| Result<T, E> 枚举，强制处理

| 学习曲线
| 平缓 (语法简洁，1–2 周上手)
| 平缓 (对现有 Java 开发者)
| 陡峭 (借用检查、生命周期、所有权模型)

| 构建工具
| go build / go install
| Maven/Gradle + native-maven-plugin
| Cargo (内置依赖管理与构建)

| 依赖管理
| Go Modules (简洁，语义化版本)
| Maven/Gradle (成熟但配置较重)
| Cargo (简洁，crates.io 生态)

| 反射/动态特性支持
| 完整支持 (reflect 包)
| 受限 (需反射配置文件，封闭世界分析)
| 编译期宏 + 有限运行时反射

| 调试与可观测性
| Delve (dlv) 调试，pprof 性能分析
| GDB/LLDB，但堆栈信息不如 JVM 友好
| GDB/LLDB，cargo-flamegraph 火焰图

| 分发与部署
| 直接下载二进制，也可 scoop/homebrew
| 直接下载二进制，但体积较大
| 直接下载二进制，cargo-binstall 一键安装

| 团队技能复用
| 中等 (后端 Go 开发者可直接上手)
| 极高 (现有 Java 团队零切换成本)
| 低 (需重新学习所有权与生命周期)

| 适用场景
| DevOps 工具、云原生 CLI、微服务客户端
| 企业级工具、需复用 Java 资产的项目
| 高性能工具、系统工具、资源受限环境

| 体积优化手段
| -ldflags="-s -w" (去符号表)，upx 压缩
| 关闭 G1GC 等特性，使用 --no-fallback
| strip 去符号表，LTO 链接优化，panic=abort
|===

== 总结建议

[cols="1,3", options="header"]
|===
| 选择 | 建议

| 选 Go
| 追求开发效率和生态成熟度，团队熟悉云原生技术栈，对 10–20MB 体积不敏感。

| 选 Java (GraalVM)
| 已有大量 Java 代码资产需要复用，团队全是 Java 背景，能接受较大的产物体积和 AOT 适配成本。

| 选 Rust
| 对产物体积、启动速度、内存占用有硬性要求，或 CLI 需要分发到资源受限环境（容器、边缘节点）。
|===
