== 资源消耗

Q: jvm 上启动一个 CuratorFramework， 添加一个 listener , 需要多少个线程？多少个tcp连接？如果有实例化两个 CuratorFramework 呢？

A: 每个 CuratorFramework 完全独立，资源线性增长。

=== 线程数

总计：约 4-6 个线程,
[source,shell]
----
1 个 CuratorFramework 实例
├── 1 个 Main-EventThread        # ZK 事件循环线程（处理 watch 回调）
├── 1 个 SendThread              # ZK 网络 IO 线程（读写 socket）
├── N 个 curator-xxx 线程        # 后台任务线程池（默认 2 个）
└── listener 回调执行            # 在 Main-EventThread 上同步执行（不另起线程）
----

如何修改 CuratorFramework 的线程数和 TCP 连接数？

[source,java]
----
CuratorFrameworkFactory.builder()
    .threadFactory(new DefaultThreadFactory("curator")) // 默认线程名前缀
    .backgroundThreads(2) // 后台线程池大小，默认 2
    .build();
----

=== TCP 连接

1 个 CuratorFramework = 1 条 TCP 长连接
但实际连接数要看 ZK 集群配置：

Curator 只和 1 个 ZK Server 建立连接
故障转移时会断开并重连到另一个 Server
不会同时连多个 Server


Q: 如果有N多 listener ,但都在  【1 个 Main-EventThread 】中同步处理，每个 listener 处理又很慢，会怎样？
A: EventThread 被阻塞 → watch 事件堆积 → ZK session 超时 → 断连重连.

正确做法：
CuratorCacheListener 里仅做异步处理 ，比如把请求转发到异步的 ExecutorService 中去处理。

== curator-xxx 后台线程池用途

主要干这些事
[source,shell]
----
curator-xxx 线程池
├── background 操作执行
│   ├── client.create().inBackground()  # 异步创建节点
│   ├── client.delete().inBackground()  # 异步删除节点
│   ├── client.setData().inBackground() # 异步写数据
│   └── client.getData().inBackground() # 异步读数据
│
├── Curator 内部组件的异步任务
│   ├── LeaderSelector 选举竞争
│   ├── DistributedBarrier 屏障同步
│   ├── 连接状态管理的重试任务
│   └── CuratorCache 的缓存预加载
│
└── ConnectionStateListener 回调
    └── 连接断开/重连/超时的回调（如果注册了的话）
----

典型场景
[source,java]
----
// 异步创建节点，不阻塞当前线程
client.create()
      .creatingParentsIfNeeded()
      .inBackground((c, event) -> {
          // 这个回调就在 curator-xxx 线程池里执行
          System.out.println("创建结果: " + event.getResultCode());
      })
      .forPath("/survival/app/cluster/ip");

// 如果你不用 inBackground()，这些线程基本闲着
client.create()
      .creatingParentsIfNeeded()
      .forPath("/survival/app/cluster/ip"); // 同步，在调用方线程执行
----

=== ZooKeeper 原生 client
ZooKeeper 原生 client（org.apache.zookeeper.ZooKeeper）不会自动重连。它的行为是：

. 会话超时后，服务端会清除该 session，所有临时节点被删除
. 客户端收到 Expired 事件后，连接进入 CLOSED 状态
. 必须由应用层手动创建新的 ZooKeeper 实例来重新连接

这就是为什么直接用原生 ZK client 时，会话超时后需要自己处理重连逻辑。


=== Curator 框架层面
Curator 会自动重连，这也是你项目中用 Curator 的原因。Curator 的 CuratorFramework 内部封装了重连机制：

* 重连策略（RetryPolicy） : 比如 ： ExponentialBackoffRetry， 这个策略的行为是：

** 连接断开后，自动尝试重连
** 重试间隔按指数退避增长（baseSleepTimeMs * multiplier^n）
** 有最大重试次数限制（maxRetries）
** 连接状态流转
Curator 的 ConnectionState 会经历以下状态：

[source,plain]
----
CONNECTED → SUSPENDED → RECONNECTED
                    ↓
                  LOST (会话超时，超过 sessionTimeout 未恢复)
----
